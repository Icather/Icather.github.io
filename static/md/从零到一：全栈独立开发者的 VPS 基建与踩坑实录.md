当一个原本纯静态的工具网站（例如基于 Cloudflare Pages 部署的语文背诵页面）开始需要状态管理——比如增加“错题本”、“星标卡片”等定制化功能时，引入后端服务和持久化存储就成了必经之路。

这篇文章记录了我将纯前端项目重构为全栈应用，并在完整 Linux VPS 环境下进行架构、网络与安全部署的真实路径。中间经历了不少新手必踩的坑，甚至遭遇了差点被黑客攻破的惊险时刻。

---

## 一、 架构抉择：分治还是聚合？

在选购 VPS 时，需求往往是矛盾的。一方面，我需要一个稳定、极速的环境来部署 Node.js + SQLite 的动态网页；另一方面，我也需要一个海外网络环境来跑 wg-easy（WireGuard 容器）以加速 GitHub 同步和个人侧的网络需求。

**试错与考量：**

最初我在几家大厂的轻量应用服务器之间横跳。国内服务器（如阿里云 99元/年 的经济型 e 实例）虽然带宽小但延迟极低，硬伤在于必须走完整的 ICP 备案流程，对于个人敏捷开发来说周期太长。而纯海外低价机器线路又得不到保障。

**破局思路：**

最终的选型逻辑回归到“合规与效率的折中”。如果追求最省心的免备案体验且兼顾国内访问，大厂（如 UCloud、腾讯云）的香港节点轻量云是最佳折中；如果死磕成本和极客体验，Oracle Cloud 的免费 ARM 实例则是终极目标。

无论最终物理机选哪种，基础运维原则必须定死：**宿主机零污染，所有服务（业务端 Node.js 与 wg-easy 代理端）全部实行 Docker 容器化物理隔离**。

## 二、 联调第一坑：Git 推送的“幽灵端口”

在本地完成前后端代码（Fetch API 对接 SQLite）改造，准备将代码推送到 GitHub 再由 VPS 拉取时，遇到了第一个极其典型的网络环境阻断问题。

**异常复现：**

执行 `git push` 时，终端直接抛出连接拒绝错误：

> `fatal: unable to access 'https://github.com/xxx': Failed to connect to 127.0.0.1 port 7890 after 2090 ms: Connection refused`

**排查过程：**

起初我怀疑是本地工作流与新买的 VPS 防火墙规则起了冲突，导致本地端口被异常阻断。但仔细审查报错日志中的 `127.0.0.1`，这实际上是本地回环地址。

打开本地代理软件的配置页面比对后真相大白：Git 客户端一直记着历史配置的代理端口 `7890`，而当前代理软件更新后，实际监听的混合代理端口变成了 `7897`。Git 正在向一个根本不存在的本地空端口发送数据包。

**应对方式：**

明确了这是教科书级的端口不匹配后，不需要去修改代理软件的底层逻辑（7897 作为混合端口同时支持 HTTP 和 SOCKS 完全够用，无需额外开启单独的 SOCKS 开关），而是应该修正 Git 的全局网络指向。

在终端执行以下命令重定向 Git 代理，问题瞬间解决：

Bash

```
git config --global http.proxy http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897
```

**工程习惯沉淀：**

解决了网络阻断后，为了保持本地与云端代码的实时感知，在 VS Code / Antigravity 环境中开启后台 `Autofetch` 选项极其关键。这能在后台静默查收云端代码变更，极大避免了在本地与 VPS 之间来回修改代码时产生的 Merge 冲突。

在解决了本地与云端的网络通信阻断后，服务初步跑通。但 Linux 服务器部署在公网上，无异于将毫无防备的婴儿扔进了黑暗森林。接下来发生的一场“虚惊”，彻底改变了我的运维安全观。

## 🕵️‍♂️ 三、 黑暗森林的敲门声：一次惊险的 SSH 溯源

项目上线几天后，我察觉到 VPS 带宽偶有异常波动。起初我天真地以为是“服务器连续开了好几天导致变慢”，试图用 `reboot` 解决问题。但在专家的建议下，我决定先对服务器进行一次深度“体检”，看看是不是遭到了外部流量的侵扰。

**异常复现：**

我通过终端执行了查看当前已建立 TCP 连接的命令：

Bash

```
ss -antp | grep ESTAB
```

返回的结果令人倒吸一口凉气。除了我自己的国内 IP 建立了正常的 SSH 连接外，赫然出现了一个**完全陌生的美国 IP (`45.78.216.103`)**，状态同样是 `ESTAB`（已建立连接），且目标直指我的 22 端口（SSH 服务）。

**诊断与溯源：**

第一反应是服务器被攻破了。我立刻提取了该连接的 PID（1550 和 1551），试图通过 `ps -p 1550 -o user,pid,etime,cmd` 查看这是什么用户在运行什么指令。紧接着，我去翻阅了系统最核心的身份验证日志：

Bash

```
grep "45.78.216.103" /var/log/auth.log
```

**戏剧性的反转出现了：**日志中并没有出现 `Accepted password`（密码验证成功）的记录。再次运行 `ss` 命令时，这个美国 IP 已经自行断开了。

**深度复盘：**

这并非一次成功的黑客入侵，而是全网自动化扫描脚本（Scanner）的一次“敲门”。它与我的服务器完成了 TCP 握手（因此显示 ESTAB），询问了 SSH 协议版本，但在尝试密码失败或根本没尝试密码后就离开了。虽然是虚惊一场，但在排查命令历史（`tail -n 500 ~/.bash_history`）时，我发现了自己留下的**致命漏洞**。

## 🛡️ 四、 亡羊补牢：告别明文与构建自动化防线

在历史记录中，我看到了自己初次部署 wg-easy 容器时敲下的命令：

Bash

```
docker run ... -e PASSWORD='我的明文密码' ...
```

**应对方式：**

这是新手极易犯的低级错误——将核心敏感信息直接暴露在 CLI 参数中，导致密码被永久写入 `.bash_history`。

1. **阻断泄露链**：第一时间重置了服务器 Root 密码，并执行 `history -c` 和 `rm ~/.bash_history` 彻底抹除历史记录文件。
    
2. **重构部署逻辑**：废弃了冗长的 `docker run` 命令，改为使用 `docker-compose.yml` 管理容器，将密码等敏感环境变量固化在配置文件中，甚至引入 `.env` 文件进行解耦。
    

## 🧱 五、 零信任基建：Fail2Ban 自动化封禁

经历了这次扫描器事件，我意识到不能让服务器处于“大门敞开任人推”的状态。最初我考虑使用 UFW（Uncomplicated Firewall）实施严苛的白名单策略（仅允许特定 IP 访问），但由于我自己的网络环境存在动态 IP 的可能，过于严苛的 UFW 极易引发“把自己锁在门外”的惨剧。

**最终方案：引入 Fail2Ban 实施动态防御。**

它能实时监控日志，当发现有人恶意猜密码时，自动调用底层防火墙将其拉黑。

1. **配置覆盖**：标准工程习惯是不直接修改 `jail.conf`，而是拷贝一份 `jail.local` 进行自定义。
    
2. **核心阈值设定**：
    
    在 `[sshd]` 模块下，我将 `maxretry` 设定为 `3`。这意味着任何 IP 只要输错 3 次密码，就会立刻触发 `bantime` 机制，被彻底关进小黑屋。
    

执行 `sudo fail2ban-client status sshd` 看到 `Total banned` 列表里逐渐增加的恶意 IP 时，终于体会到了一丝运维层面的安全感。

## 🚀 六、 写在最后：从“能跑就行”到“工程直觉”

这次将背诵网站迁移至独立 VPS 的经历，不仅让我打通了从前端页面、Fetch API 联调到 SQLite 持久化存储的全栈闭环，更是一次对 Linux 底层网络与防御机制的实战演练。

当代码离开 Localhost 暴露在公网的那一刻起，“能跑就行”的思维就必须让位于“安全合规与架构稳健”。每一次的报错拒绝、每一个异常的 TCP 连接，最终都化为了全栈开发路上不可或缺的工程直觉。
